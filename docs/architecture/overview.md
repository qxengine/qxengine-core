## What is QXEngine?

QXEngine is a portable, C-ABI memory-intelligence engine designed for
mobile and cross-platform applications. It provides deterministic memory
allocation, constitutional law enforcement, cognitive snapshot capture,
and real-time telemetry — all through a single opaque handle exposed as
a plain C API.

The engine is written in C++17 internally but exposes no C++ symbols
across its boundary. Every public function is `extern "C"`, making it
consumable from Swift, Kotlin (via JNI), Python (via ctypes), and any
other language with a C FFI.

---

## Design Principles

| Principle | Description |
|---|---|
| **C ABI boundary** | No C++ types cross the public API. All handles are opaque pointers. |
| **Constitutional laws** | Eight immutable laws govern every allocation, evaluation, and snapshot. |
| **Immutable audit trail** | Law reports and telemetry events are append-only ring buffers. |
| **Deterministic limits** | Soft and hard memory limits are enforced per-segment, not globally. |
| **Native-first policy** | Law 6 mandates that native allocations take priority over interpreted layers. |
| **Zero hidden globals** | All state is owned by the engine handle. No process-wide singletons. |
| **Thread safety** | All post-create functions are thread-safe. Create and destroy are not. |

## Layer Breakdown

The engine is organized into five layers, each with a strict
single responsibility.

**Layer 1 — Core types and constants** (`include/qxengine/core/`)
Defines all primitive types (`QXError`, `QXTierId`, `QXEvictClass`),
compile-time constants (`QX_SEGMENT_COUNT = 9`, `QX_LAW_COUNT = 8`),
and the version header generated by CMake at build time.

**Layer 2 — Memory subsystem** (`include/qxengine/memory/`)
Implements the leaf-segment-memloc hierarchy. A `QXLeaf` is the atomic
allocation unit. Nine `QXSegment` instances form a `QXMemloc`. The
`QXPressureCoordinator` monitors utilisation across all segments and
triggers eviction when thresholds are breached.

**Layer 3 — Law subsystem** (`include/qxengine/law/`)
The `QXLawEnforcer` evaluates all eight constitutional laws against a
`QXLawEvaluationInput` snapshot and produces a `QXLawReport` containing
per-law scores, a composite health score (0–100), a certification tier,
and an immutable audit history capped at 1 000 entries.

**Layer 4 — Intelligence subsystem** (`include/qxengine/intelligence/`)
The `QXSnapshotHistory` captures `QXCognitiveSnapshot` records at
configurable intervals. Each snapshot computes a knowledge score from
five weighted signals: memory clarity, pressure stability, law
compliance, native coverage, and recency.

**Layer 5 — Manifest subsystem** (`include/qxengine/manifest/`)
The `QXManifest` parses and validates the application JSON manifest
against 14 MFT rules before the engine is created. The manifest is
immutable after a successful parse.

## Data Flow

A typical allocation and evaluation cycle proceeds as follows:

1. **Parse manifest** — `qx_manifest_parse()` validates the JSON and
   returns an immutable `QXManifestHandle`.
2. **Create engine** — `qx_engine_create()` initialises all six
   subsystems in dependency order, wiring the manifest limits into
   the memory subsystem.
3. **Allocate** — `qx_engine_alloc()` routes the request to the
   correct segment, creates a leaf, and records a telemetry event.
4. **Evaluate** — `qx_engine_evaluate()` gathers a
   `QXLawEvaluationInput` from the memory subsystem, runs all eight
   laws, caches the report, and records telemetry.
5. **Snapshot** — `qx_engine_snapshot()` computes the five cognitive
   signals and appends a `QXCognitiveSnapshot` to the history ring.
6. **Free** — `qx_engine_free()` marks the leaf as released, updates
   counters, and records telemetry.
7. **Destroy** — `qx_engine_destroy()` tears down subsystems in
   reverse order. The caller destroys the manifest separately.

## Threading Model

| Function group | Thread-safe |
|---|---|
| `qx_engine_create` / `qx_engine_destroy` | ❌ Not thread-safe |
| `qx_manifest_parse` / `qx_manifest_destroy` | ❌ Not thread-safe |
| All other `qx_engine_*` functions | ✅ Thread-safe (internal mutex) |
| All subsystem accessors after create | ✅ Thread-safe |

The engine uses one mutex per subsystem handle. Evaluation, allocation,
and snapshot capture can be called concurrently from multiple threads
after a successful `qx_engine_create`.
